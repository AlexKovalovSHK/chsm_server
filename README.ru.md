# CHSM Classroom Integrations

CHSM Classroom Integrations — это backend API сервис, написанный на **NestJS**. Он выступает в роли центрального узла, объединяющего Google Classroom, Telegram и пользовательские frontend-приложения/расширения в единую систему управления пользователями и уведомлениями.

## 📌 Обзор системы

Основная цель этой системы — управление учениками и преподавателями на разных платформах с синхронизацией данных между Google Classroom и Telegram. Она позволяет администраторам отслеживать учеников, синхронизировать состояние курсов и отправлять массовые объявления или личные сообщения через Telegram-бота.

### Ключевые возможности
1. **Управление пользователями (DDD Паттерн)**:
   - Полный CRUD для пользователей с ролями (`student`, `admin` и др.).
   - Хранение Telegram ID (`tgId`) и Google ID (`googleId`) для объединения учетных записей.
2. **Управление студентами (DDD + Onion Архитектура)**:
   - Четкое разделение ответственности между слоями Domain, Application, Infrastructure и Interface.
   - Независимость от персистентности (Prisma) через интерфейсы репозиториев.
3. **Интеграция с Google Classroom**:
   - Авторизация через OAuth2.
   - Синхронизация активных курсов, учеников и учителей.
   - Получение заданий и оценок.
   - Возможность отправки глобальных объявлений во все курсы Classroom.
   - Отправка email-сообщений через Gmail API.
4. **Интеграция с Telegram Bot**:
   - Построено на `grammy` в качестве внутреннего API для Telegram-бота.
   - Предоставляет эндпоинты для синхронизации пользователей Telegram, обновления шагов регистрации и блокировки.
   - Позволяет рассылать сообщения конкретным пользователям Telegram по `tgId` или `email` (используется Chrome расширением).
5. **Аутентификация и безопасность**:
   - Аутентификация на основе JWT с использованием Passport.
   - Защита документации Swagger с помощью Basic Auth.
   - Настроенный CORS для работы с административными панелями и браузерными расширениями.

## 🏗 Архитектура

Проект структурирован в соответствии с принципами **Domain-Driven Design (DDD)** и модульной архитектурой NestJS:

```text
src/
├── prisma/                 # PrismaService и глобальный клиент БД
├── app.module.ts           # Корневой модуль, подключающий Postgres, планировщик и все домены
├── auth/                   # Домен аутентификации (JWT, Guards, Стратегии)
├── classroom/              # Логика интеграции с Google Classroom
│   └── service/            # Бизнес-логика для Google API (курсы, Gmail, студенты)
├── telegram/               # Логика Telegram-бота и обмена сообщениями
│   ├── controller/         # Webhooks и внутренний REST API (например, рассылки)
│   └── service/            # Взаимодействие с Bot API и логика синхронизации пользователей
├── students/               # Домен студентов (DDD + Onion Архитектура)
│   ├── domain/             # Сущности и интерфейсы репозиториев
│   ├── application/        # Use Cases, сервисы приложения и DTO
│   ├── infrastructure/     # Реализация персистентности (Prisma) и мапперы
│   └── interface/          # HTTP контроллеры и точки входа
└── users/                  # Основной домен пользователей
    ├── application/        # Сервисы приложения и DTO (UserService, Mappers)
    ├── controllers/        # REST API для пользователей
    └── infrastructure/     # Логика БД (Prisma UserRepository)
```

### Стек технологий
- **Фреймворк**: NestJS (TypeScript)
- **Среда выполнения**: Bun
- **База данных**: PostgreSQL (Prisma 5.x)
- **API**: Google APIs (Classroom, Gmail, OAuth2), Telegram Bot API (`grammy`)
- **Контейнеризация**: Docker и Docker Compose (образ `chsm_serv` работает на порту 5008)

## ⚙️ Принцип работы

1. **Процесс аутентификации**:
   - Пользователи/Админы могут авторизоваться через стандартный логин (получая JWT) или через Google OAuth.
   - При авторизации через Google система получает токены и профиль Google, связывая их с существующим email или Telegram ID.

2. **Синхронизация данных**:
   - **Telegram**: Telegram-бот вызывает `POST /internal/users/upsert` или `PATCH /internal/users/sync` для создания или обновления пользователей в базе данных PostgreSQL.
   - **Classroom**: Администраторы могут загрузить последний список учеников, учителей и активных курсов напрямую из Google API. Вся персистентность проходит через `UserService`.

3. **Механизм рассылок (Бродкаст)**:
   - Система открывает эндпоинты вроде `POST /internal/users/broadcast-extension`.
   - Chrome расширение или Admin Web Panel отправляет список email-адресов и текст сообщения.
   - Система ищет соответствующий `tgId` для каждого email и отправляет сообщение через Telegram Bot API (`BotApiService`).

## 🚀 Быстрый старт для Агентов/Разработчиков

### Требования
- Установленный Node.js (или Bun).
- База данных PostgreSQL.
- Файл `.env` с учетными данными Google API (`GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GOOGLE_REDIRECT_URI`), токеном Telegram-бота, JWT секретом и `DATABASE_URL`.

### Запуск проекта локально

```bash
# Установка зависимостей
npm install (или bun install)

# Запуск в режиме разработки с использованием Bun
npm run start:dev
```

### Запуск с помощью Docker

```bash
docker-compose up -d --build
```

### Важные API Эндпоинты
- **Документация Swagger**: `/api/docs` (Защищена Basic Auth: `admin` / `Abc!1234`)
- **Google Auth**: `/classroom/auth`
- **Рассылка по пользователям**: `/internal/users/broadcast-extension` (Ожидает массив email и текст)
- **Telegram Webhook/Внутренние API**: `/internal/users/...`
- **Журналы практик**: `/practices`
  - CRUD операции для журналов практик.
  - `GET /practices?enrollmentId=` для получения всех журналов практик для зачисления.
- **Записи практик**: `/practices/:practiceId/entries`
  - CRUD операции для отдельных записей практик внутри журнала.
  - `PATCH /practices/:practiceId/entries/:id/approve` для одобрения записей.

### Контекст для дальнейшей работы
При работе с агентами в этом репозитории:
- Любые новые модели баз данных должны размещаться в папках `infrastructure` или `schemas` соответственно.
- Бизнес-логику следует сохранять в слоях `application` или `service`.
- Будьте осторожны с изменениями CORS в `main.ts`, так как Chrome расширение и панели администратора зависят от конкретных origin.

### Технические вопросы

- Бекенд и фронтенд находятся на VPS IONOS (большой) 
- Основная база данных на VPS IONOS (маленький) в Docker
- Резервная база данных на Neon (работает не очень стабильно, но бесплатно)
